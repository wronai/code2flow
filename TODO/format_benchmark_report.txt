RECOMMENDATION — Unified Format for Data-Flow Refactoring
══════════════════════════════════════════════════════════════════════════════════════════

  None of the 3 formats alone is sufficient for data-flow refactoring.
  Each has critical gaps that the others fill:

  ┌─────────────────────────────────────────────────────────────────────┐
  │  IDEAL FORMAT = merge of strengths from all three:                 │
  │                                                                     │
  │  From project.toon:   Type signatures, import graph                │
  │  From analysis.toon:  HEALTH, REFACTOR, call chains, CC metrics    │
  │  From llm_prompt.md:  Behavioral patterns, docstrings, API surface │
  │                                                                     │
  │  Plus NEW (missing from all):                                       │
  │    • DATA_FLOW section: explicit pipeline stages                   │
  │    • TRANSFORMS: what each function does to data                   │
  │    • CONTRACT: input type → output type per function               │
  │    • PIPELINES: automatically detected A→B→C chains               │
  └─────────────────────────────────────────────────────────────────────┘

  Proposed naming and file structure:

    output/
    ├── analysis.toon      ← KEEP: the health/diagnostic format (renamed from 'toon v2')
    ├── project.map        ← RENAME: structural map (was project.toon)
    ├── context.md         ← RENAME: LLM narrative (was llm_prompt.md)
    └── flow.toon          ← NEW: data-flow-focused format (proposed below)

  The NEW flow.toon format should focus on DATA FLOW:

  ```
  # code2flow/flow | 335 functions | 12 pipelines | 2026-03-01

  PIPELINES[4]:
    NLP: raw_query → normalize(str→str) → match_intent(str→IntentMatch)
         → resolve_entities(IntentMatch→Entity[]) → format_action(Entity[]→str)
    Analysis: path → collect_files(Path→Path[]) → analyze_file(Path→AnalysisResult)
              → merge_results(AnalysisResult[]→AnalysisResult) → build_call_graph
    Export: AnalysisResult → compute_metrics → render_sections(ctx→str[]) → write(str)
    Refactoring: AnalysisResult → detect_smells → build_context → render_prompt(→str)

  TRANSFORMS (sorted by fan-out):
    main(args:list) → {Config,Path,AnalysisResult}  fan=45  !! God Function
    _compute_file_metrics(result:AR) → Dict[str,FileMetrics]  fan=22
    analyze_project(path:Path) → AnalysisResult  fan=18  PIPELINE:Analysis.entry
    process(query:str) → NLPPipelineResult  fan=18  PIPELINE:NLP.entry

  CONTRACTS (input→output per function, grouped by pipeline):
    Pipeline: NLP
      normalize(query:str, lang:str) → str           CC=6   pure
      match_intent(normalized:str) → IntentMatch      CC=7   pure
      resolve(match:IntentMatch) → Entity[]           CC=13  side-effect:cache
      format_action(entities:Entity[]) → str          CC=10  10 branches

  DATA_TYPES (ranked by usage across functions):
    AnalysisResult    consumed:18  produced:4   hub-type
    str               consumed:45  produced:30  ubiquitous
    Path              consumed:12  produced:3   input-heavy
    FunctionInfo      consumed:8   produced:2   read-mostly
  ```

  This format enables refactoring by DATA FLOW:
  - 'AnalysisResult consumed:18' → too many dependents, split the type
  - 'Pipeline NLP is pure' → safe to parallelize
  - 'main() → 45 outputs' → not a function, it's a script in disguise
  - 'resolve() has side-effect:cache' → mark as impure in pipeline
